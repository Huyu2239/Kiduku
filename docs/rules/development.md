# 基本方針

## 目的

本リポジトリでの開発判断の基準を明確にする。

## 全般方針

- Rust の公式な慣習に従う
- 単純で直線的な構造を維持する
- 不要な抽象化・不要な最適化は行わない
- 可読性と保守性を優先する
- 変更は小さく、意図が追える粒度に保つ

## MVP 開発における方針

### アーキテクチャ

- **Presentation層が Discord 操作を集約**: Serenity API のすべての操作は Presentation 層で行う
- **Usecase層は純粋ロジック**: ビジネスロジックのみを扱い、外部API呼び出しは行わない
- **DTO形式で入出力統一**: Usecase の入出力は必ず DTO 形式にする
- **ポートは不要（Phase 2以降で導入）**: MVP では外部依存の抽象化（trait）は避ける

### テスト

- Domain/Usecase は単体テスト推奨（外部依存がないため）
- Presentation は E2E テスト推奨（Discord テストサーバーでの手動確認）

### 実装の進め方

- 優先度は `/help` → `/check-reads` → 自動リアクション付与
- 各フェーズは独立性を保つ（Phase 1 完了後に Phase 2, 3 並列実装可能）
